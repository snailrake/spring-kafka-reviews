## Переменные и типы
- Всегда объявляй локальные переменные через `var`.

## Возвраты из методов
- Предпочитай **один `return` на метод** (единая точка выхода).
- Строй код так, чтобы финальный результат возвращался в конце метода.
- Не вводи промежуточную переменную только ради немедленного возврата.
  - Правильно: `return ResponseEntity.ok(body);`
  - Неправильно: `var result = ResponseEntity.ok(body); return result;`

## Циклы
- В циклах старайся не использовать `break` и `continue`.
- Вместо этого используй:
  - корректную структуру условий (`if/else`)
  - вынос логики в отдельные методы
  - предварительную фильтрацию коллекций, если это уместно

## Null и валидации
- Для проверок используй статические методы классов.
- Не используй полные квалифицированные имена классов в теле кода вместо импортов.
  - Правильно: импорты сверху, использование коротких имен в коде.
  - Неправильно: `java.util.Objects.isNull(x)` в теле методов.
- Для строковых проверок предпочитай готовые методы `StringUtils`, например `isBlank()` / `isNotBlank()`, вместо ручных комбинаций `null`, `isEmpty()`, `trim()` и подобных.
- В проверках в целом старайся использовать минимально необходимое количество методов и условий, если это не ухудшает читаемость.

## Optional
- Если метод по логике может "ничего не вернуть", предпочитай `Optional` вместо `null`.
- Не возвращай `null` там, где можно вернуть `Optional.empty()`.

## DTO
- DTO по возможности делай через `record`.
- Если DTO требует мутабельности или специфики фреймворка, допускается обычный класс как исключение.

## Импорты
- Все импорты должны быть сверху файла, по стандарту.
- Не используй "прямые импорты" в коде.
- Не используй полные пакеты в теле методов.
- Не используй `static import` для методов в прод-коде.
- В тестах `static import` разрешен, например, `AssertJ` / `Mockito` / `JUnit`.
- Вызывай статические методы через имя класса, чтобы в коде было видно источник вызова.
  - Правильно: `Objects.isNull(x)`
  - Неправильно: `isNull(x)`

## Логирование
- Логи добавляй только там, где это действительно нужно: интеграции, границы системы, ошибки, важные ветки и т.д.
- В тестах логи по умолчанию не добавляй.
- Не добавляй "шумные" логи без явной пользы.

## Комментарии
- Не писать комментарии в коде, если об этом не просили явно.

## Тесты
- Для тестовых классов использовать `@DisplayName` в формате: `Тест для <ClassName>`, например: `Тест для ProxyService`.
- Для тестовых методов использовать `@DisplayName` в формате: `Когда ... тогда ...`, например: `Когда inputInvalid_thenThrows()`.
- Имена тестовых методов писать в формате: `<methodName>_when..._then...`, например: `process_whenInputInvalid_thenThrows()`.
- Внутри тестовых методов разделять блоки комментариями `// Given`, `// When`, `// Then` допускается краткие варианты в любом сочетании, например `// Given / When`, `// When / Then`, `// Given / Then`.
- Интеграционные тесты именовать с суффиксом `IntegrationTest` и запускать отдельной gradle-задачей `integrationTests` (по умолчанию не входят в `test`).
